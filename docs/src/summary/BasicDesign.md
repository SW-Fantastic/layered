# 基本设计

这个项目的基本目的是通过metadata实现各种类库的跨语言调用，为此我有如下的设计：

## 解析头文件生成Metadata

从头文件获取本地库的元数据毫无疑问是最准确的，我发现libclang可以做到这一点，而且libclang
具有C语言形式的接口，而且它是老牌的C语言编译器项目，因此成为了我实现Metadata解析的首选项。

通过JavaCpp可以较为轻松的把libclang接入Java环境，从而实现解析，我的另一个项目已经完成了这个
准备工作。

解析头文件是一个较为繁琐的过程，我计划从C语言出发，首先实现基础的解析验证本项目的可行性，然后逐步
拓展到C++，因此项目的第一阶段是实现一个C语言库的header解析功能，解析头文件的各个组成部分并且输出
完整的元数据。

得到元数据后，下一步是整理这些元数据，将所有的内容都收缩到这样的范围，所有对象（Struct）都必须以
指针的形式传递，所有的基本类型都需要是有符号的，但是我不打算进行数值上的转换，只是单纯将它们`视作`
有符号的形式，例如通过`static_cast`。

这种做法主要是为了兼容Java，目前的Java语言不支持无符号基本类型。

## 数据结构的扁平化和函数化

虽然按照内存的Offset直接读写本地对象的字段是一种很不错的做法，但是，由于内存对齐这种特性的
存在，如果想要将Class或者Struct与Java或者其他语言的字段一一对应，就得手动处理这些内存的Padding，
很麻烦，容易出错。

我想到了Java是有getter和setter的，为这些本地字段提供一个本地的get或者set方法应该非常不错，
同时，构造方法和析构方法如果存在也应该得到解析，并且生成到胶水代码中。

对于C或者C++来说，Object的创建分为两部分，一个是内存的分配，一个是对象的初始化，`new`运算符
可以按照合理的顺序完成这一步骤，对于一个这样的本地对象，它的尺寸是被硬编码到二进制组件中的，所以，
我们没有办法在生成的库中获取到一个本地对象究竟有多大，所以，即使我可以得到它的构造方法和析构方法，
也无法可靠的构建出一个本地对象。

我知道有一些项目非常的`Hack`，它们可以逆推出来一个本地对象的大小，但是这不能确保每一个类型的本地对象
都能准确无误的获取到，这也是为什么我最终选择生成胶水代码，其一是因为我想要统一参数和返回值到基本类型，
其二是因为我想要本地函数用作Getter和Setter，其三就是因为这样可以让对象的构建变得可靠。

## 代码和项目的生成

我认为CMakeList是一个非常不错的C语言构建工具，因此，我会采用CMake构建生成的胶水代码以及用于本项目的
编译和生成。

以此为目标，C/C++的头文件解析完毕后，将会生成C语言的header和源码，并且输出一个CMake项目。

为了方便的引用C/C++的开源组件，本项目还需要使用`vcpkg`。
