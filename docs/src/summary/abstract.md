# Layered是个什么项目

本项目是一个基于libclang构建的外部接口运行时，它的主要目标是把C语言，C++语言的开源项目，
以其header为基础整理为标准的C语言接口，以便于跨语言的调用。

## 本地库相互调用的主要阻碍

如果我们想要在另一个语言中使用本地库（例如C/C++），那么就需要获取到动态库的符号表（Symbol table），
通过符号表我们就可以查找到想要调用的函数的内存地址，最开始C 或者类似C语言的本地语言导出的符号表非常简单，
它只包含函数的名称，但是后来随着面向对象编程逐渐 成为主流，类，接口，重载这些新概念的引入，只有函数名称对于符号表来说是不够的。

所以，这就出现了一个新概念——Mangled Name。

## 名称混淆（Name Mangling）

所谓的MangledName，简单来说是把其他的必要信息也整合到函数的导出名称中，例如函数的参数表
等，不同的C++编译器导出它们的规则是不同的，因此，一个库如果是使用微软的Microsoft MSVC进行编译
，就很可能无法和使用GNU编译器的项目进行链接，因为GNU无法解析出正确的MSVC函数。

那么为什么编译器需要这样一个步骤，把名称搞成看不懂的样子呢，这其实是因为很多语言本身的特性会导致出现
不同函数使用同一个名称的问题，例如C++类的函数重载，它的函数名称是相同的，不同的只有参数，因此只导出
函数名称就会导致一个函数名称对应多个函数的情况，这就是所谓的符号冲突。

顺便一提，其他的编译型语言通常也有自己的Name Mangling规则，例如Rust，Go等，但这里不作为重点。

C语言在所有编译器中是一个例外，MSVC和GNU编译的C语言库非常通用，或者说，所有的C语言库几乎都是通用的，因为C语言
导出符号的规则非常简单，只导出函数名，没有任何额外的信息。

在MangledName之外，阻止语言之间相互调用的还有调用约定。

## 调用约定（Calling Convention）

调用约定，它讲的是库函数应该如何调用，有人会感到奇怪，“什么叫函数应该如何调用”，函数的调用呢，
其实和我们印象中有所不同，一个库被加载后，我们可以得到符号表，解析符号表可以得到函数的名称以及函数的内存地址，
，如果这一种语言还有其他的东西在包含在其中（Mangled Name），那么调用的时候这些都会被解析到。

这些信息里面，最重要的就是函数的内存地址，通过这个地址，就能够调用这个函数从而实现功能，想要调用一个函数
就需要知道我们应该把参数放在哪里，按什么顺序放，调用完毕之后，这些参数应该如何清理以回收资源，而这些，就是调用
约定的内容。

## 本项目要做什么

本项目想做的，就是想办法抹平Name Mangling和Calling Conversation等差别，把C/C++的本地库全部统一为扁平的，
基于基本类型和指针的，包含完整的Metadata的，且采用自定义的Name Mangling规则的进行查询和调用的全新外部调用体系。
